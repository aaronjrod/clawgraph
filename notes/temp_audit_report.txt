2. Major Architectural Gaps Discovered
The audit revealed two significant gaps where the clawgraph/orchestrator/ implementation falls short of the behavioral specifications.

Gap 1: Prerequisite Re-evaluation Pipeline (
05_ARCHITECTURE.md
 S10.2 / 
03_FRS.md
 F-REQ-34)
Specification: Nodes missing prerequisites (
requires
) should emit a STALLED state and be placed in a queue. Every time a node emits DONE, the Orchestrator must enter a RESOLVING state, scan the STALLED queue, re-check prerequisites against the document_archive, and move satisfied nodes to the READY queue.
Implementation Reality: In 
hub.py
 (
_make_dispatch_node
), if prerequisites are missing, the Orchestrator emits a STALLED observability event to the Signal Manager, but immediately fails the current execution by returning a synthesized NEED_INTERVENTION ClawOutput. There is no queue, no RESOLVING state, and no re-evaluation loop. The job is effectively terminated if a dependency isn't met at the exact moment of dispatch.
Gap 2: HITL Timeline Context Injection (
03_FRS.md
 F-REQ-32)
Specification: When a HOLD_FOR_HUMAN signal is emitted, the Orchestrator must automatically package the preceding $N$ events (default 5) from the Timeline into the intervention payload to provide lead-up context to the human.
Implementation Reality: In 
hub.py
 (
_make_suspend_node
), the state is correctly checkpointed and the registered 
hitl_handler
 is called. However, it only passes the 
human_request
 dictionary from the node's output. It fails to query the TimelineBuffer.get_hitl_context() and does not inject the historical context into the payload sent to the handler.

3. Additional Architectural Gaps Discovered
Gap 3: Missing Escalation Policy Enforcement (
03_FRS.md

Specification: Nodes emitting NEED_INFO must have their escalation_policy (TTL and max retries) respected. Upon budget exhaustion, the Orchestrator must automatically promote the signal to NEED_INTERVENTION.
Implementation Reality: In

Gap 4: Incomplete PARTIAL Signal Semantics (
03_FRS.md

Specification: A PARTIAL signal MUST include a

Implementation Reality:
The

The Orchestrator's

partial_commit_policy
 subgraphs.
Gap 5: Multi-Domain Tag Visibility Enforcement (
03_FRS.md

Specification: The document_archive must support multi-bag tagging to allow artifacts to be visible across sovereign workspaces while preventing unauthorized cross-talk.
Implementation Reality: The document_archive in

) is implemented simply as dict[str, str] mapping IDs to URIs. There is no tagging system implemented, and no visibility boundaries enforced when the Orchestrator checks prerequisites.
Action Items from Part 3
Implement TTL/Retry Monitors: Add state tracking variables to

Reconcile PARTIAL Schema: Move

Part 4: Business Requirements Audit (02_BRS.md, 01_PLAN.md)
The final audit sweep reviewed the high-level business logic and core product philosophy documents to ensure the technical implementation aligns with the original vision.

6. Architectural Alignment & BRS Correlations
The gaps identified in Parts 2 and 3 trace directly back to explicit Business Requirements that are currently unfulfilled by the orchestrator:

B-REQ-14 (Sovereign Dependency Resolution): The FRS STALLED state gap (Gap 1) directly violates B-REQ-14, which mandates that the Orchestrator must mark nodes as STALLED and prioritize producer nodes, rather than failing the execution.
B-REQ-15 (Super-Orchestrator Stalemate Intervention): Because the RESOLVING loop is missing (Gap 1), the Super-Orchestrator currently has no mechanism to "prime the state" and force unblock a STALLED queue as required by B-REQ-15.
B-REQ-18 (Contextual Intervention): The missing timeline injection in 
_make_suspend_node
 (Gap 2) directly violates B-REQ-18, which strictly requires human reviewers to be provided with the "lead-up" historical events to ensure sufficient grounding.
Conclusion: The core node models and Super-Orchestrator prompting logic are exceptionally well-aligned with the specs. However, the Orchestrator runtime (
hub.py
/
graph.py
) requires a significant refactor of its 
dispatch
 and 
route_signal
 logic to fully realize the dynamic, dependency-resolving, and policy-enforcing vision outlined in the documentation.

Part 6: Deep Structural Audit (Round 5)
Per request, a 5th deep audit sweep was conducted against the FRS and Architecture documents focusing on the Orchestrator's core event loop. This revealed a massive, foundational discrepancy preventing the framework from operating as designed.

Gap 6: Single-Node Execution Limit (Violates F-REQ-12)
Spec (F-REQ-12: Hub-and-Spoke Routing): "Finished nodes shall return control and a payload to the Orchestrator (the Hub), which determines the next activation based on the signal." Implementation Defect: The hardcoded routing logic in hub.py::route_signal returns ROUTE_COMPLETE upon receiving any DONE signal:

python
if signal == Signal.DONE:
        return ROUTE_COMPLETE
Because the Orchestrator does not track a completed_nodes set or a READY queue (it only tracks current_node_id), and because DONE unconditionally terminates the LangGraph execution string, the Orchestrator can only physically execute a maximum of 1 node per 
start_job
 call. A multi-node "Bag of Nodes" is structurally impossible to iterate through under the current logic.

Action Required:

Add READY, STALLED, and COMPLETED queues to 
BagState
.
Rewrite 
route_signal
 to return ROUTE_NEXT_NODE on DONE if the READY queue is not empty.
Rewrite 
_make_dispatch_node
 to pop the next available node from the READY queue instead of statically looking at current_node_id.
